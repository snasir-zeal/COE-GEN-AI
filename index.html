The problem with "agentic"

More agents
doesn't mean better.

The industry's default playbook: give the LLM tools, let it decide when to call them, chain agents together, and hope for the best. Tool-calling agents orchestrating other tool-calling agents. It sounds powerful. In production, it's fragile.

One misinterpretation anywhere in the chain triggers bad effects exponentially. It's the same problem we see with agentic coding today — one bad plan line produces hundreds of bad code lines, and the next LLM uses that bad context to plan the next step. The errors compound. You lose control.

The design philosophy

LLMs do language.
Python does logic.
SQL does data.

Use each tool where it's strongest. LLMs are exceptional at planning, classification, natural language parsing, and reasoning. They're terrible at orchestration, calculation, SQL generation, and deterministic workflows. So don't ask them to do those things.

LLMs
WHAT THEY'RE GOOD AT
Intent classification. Slice planning — decomposing a complex question into structured sub-queries. Natural language reasoning over computed results. Synthesizing insights from data.
Python
WHAT CODE IS GOOD AT
Orchestration. Workflow control. Semantic resolution via fuzzy matching. Query building from templates. Calculations — elasticity, lift, cannibalization. Retry logic. Error handling.
SQL
WHAT DATA ENGINES ARE GOOD AT
Data retrieval. Aggregation. Filtering. Joining. The actual heavy lifting. Templated queries ensure correct, performant SQL every time — no hallucinated column names.
The rule
If Python can do it deterministically, don't give it to an LLM. If a SQL template can generate the query, don't let the model write SQL. Reserve LLM calls for what only language models can do: understanding ambiguity, planning approaches, and reasoning over results.
The client

Sunset Foods.
Real data, real questions.

A regional grocery chain on Chicago's North Shore. Seven stores, 63 million raw transactions spanning 2023 through 2025. The kind of data density where the right question — answered fast — changes how you stock shelves, price products, and run promotions.

Analysts and marketing teams needed to ask proprietary questions about their data in natural language: Did this promo cannibalize that product? Was the lift from the sale or the holiday? What sells with rotisserie chicken, and how strong is that relationship?

The challenge
Build an analytics agent that non-technical users can ask complex, multi-part questions — and get answers they can trust, backed by real SQL against real data.

System architecture

One app. One tool. One workflow.

A FastAPI application on Azure App Service serves both the chat frontend and the backend. The conversation agent has a single tool: a workflow graph built on Microsoft Agent Framework. No chains of agents. No tool-selection ambiguity. One structured pipeline that always runs the same way.

The spine of the system

WorkflowState:
typed, growing, traceable.

A single Pydantic model passes across every node in the DAG. Each step reads what it needs from previous steps, does its work, and writes its output back. By the end, WorkflowState has the full trace — from raw question to reasoned answer.

This is what makes the system debuggable. Every field is typed. Every step's contribution is visible. You can inspect the state at any point and know exactly what happened, what was planned, what SQL ran, and what the reasoner received.

Full traceability — every LLM output, every SQL query, every calculation result is captured in typed fields.
No hidden state — no agent memory, no implicit context. Everything is explicit in the model.
Composable nodes — each DAG node is just Python code that reads from and writes to WorkflowState.

No hallucinated SQL

The Semantic Registry:
business knowledge in YAML.

Instead of hoping the LLM knows your database schema, encode it explicitly. Every department, category, store, and timeframe is mapped in YAML with synonyms for fuzzy matching. When the Slice Planner says "sausage," deterministic Python resolves it to the exact database value "SAUSAGES" — no LLM involved.

How resolution works
LLM says "bratwurst"
→
Fuzzy match → "SAUSAGES"
→
SQL: WHERE Category = ?
Why this matters
An LLM writing raw SQL would guess column names, misspell categories, forget which store is "Store 001," and produce queries that silently return wrong data. The registry makes resolution deterministic and auditable.
# ── CATEGORY GROUPS ── sausage: value: "SAUSAGES" synonyms: - "sausage", "sausages" - "breakfast sausage" - "italian sausage" - "bratwurst", "kielbasa" hummus: value: "RF HUMMUS" synonyms: - "hummus", "humous", "houmous" # ── STORES ── highland_park: value: "Store 001 - Highland Park" match_pattern: "%Highland Park%" store_number: "001" synonyms: - "store 1", "highland", "hp" long_grove: value: "Store 006 - Long Grove" match_pattern: "%Long Grove%" store_number: "006" synonyms: - "store 6", "lg", "long grove"
Templated, not generated

Four SQL templates
cover every use case.

The LLM never writes SQL. Instead, the Slice Planner outputs a structured plan — which template to use, which group-by columns, which metrics, which filters. Then deterministic Python reads that plan, resolves filters against the semantic registry, and builds the query from a template.

Aggregation
Totals, rankings, comparisons, grouped summaries. The workhorse — handles most questions.
Time Series
Trends over time. Weekly sales, monthly units, daily price movements.
Market Basket
What sells together. Basket-level analysis for association and halo effects.
Lookup
Simple data retrieval. Product lookups, store info, raw record checks.
The contract
The planner creates the contract — what data to get and how. The runner fulfills it deterministically. The LLM never touches SQL, and the SQL never depends on the LLM.

The pipeline in action

Five nodes. Three LLM calls. Zero ambiguity.

Each node in the DAG is just Python code. It reads from WorkflowState, does its job, writes back, and passes it along. LLMs are called only where natural language understanding is required — everywhere else, it's deterministic.

1
Intent
LLM
Classifies the user's question. Is it an analytics question or a basic lookup? Extracts entities mentioned.
Reads: user_question
Writes: intent
2
Planner
LLM
Decomposes the question into structured slices. Each slice specifies template, filters, metrics, group-bys, and calculator requests.
Reads: intent
Writes: slice_plan
3
Runner
PY
Loops through each slice. For each: resolves filters via semantic registry, builds SQL from template, executes query. All deterministic.
Reads: slice_plan
Writes: dataset_refs
4
Calc
PY
Runs requested calculators: price elasticity, lift analysis, cannibalization, period comparison, halo effect. Pure Python math.
Reads: dataset_refs
Writes: calculation_results
5
Reasoner
LLM
Receives the question, all plans, all data summaries, and all calculation results. Synthesizes into business insights. This is what LLMs are built for.
Reads: everything
Writes: reasoner_output
The count
Three LLM calls total. Intent, Plan, Reason — the three things only a language model can do. Everything between is Python. No agent decides whether to call another agent. No tool-selection ambiguity. The workflow is the same every time.
The final mile

Give the LLM everything. Let it reason.

By the time we reach the Reasoner, the hard work is done. The LLM doesn't need to pull SQL — we've already given it everything: the original question, every slice plan, every dataset summary, every calculation result. It just needs to think.

The design philosophy

Less agentic.
More reliable.
Actually works.

The industry will keep chasing more agents, more tools, more orchestration layers. The systems that actually make it to production — the ones that handle 63 million transactions and answer questions people stake decisions on — are the ones that use AI where it's brilliant and code where it's better.
